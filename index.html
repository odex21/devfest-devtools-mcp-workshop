GDG 与 DevFest：开发者的线下能量场

Google Developer Groups（GDG）是面向开发者的本地技术社区，而 DevFest 则是 GDG 社区一年一度的技术盛会。 这里不仅有分享和 Workshop，更重要的是能遇到一群同样热爱技术的人。

在 GDG 社区，你会发现很多熟悉的关键词： Android、Web、Cloud、AI，也会发现很多不那么陌生的人—— 大家来自不同的公司、学校和城市，但在 DevFest 的现场，大家有一个共同的身份： 「热爱分享与学习的开发者」。

这次 Workshop 以「Chrome DevTools MCP Server」为主题，从浏览器调试到自动化、从本地开发体验到 MCP 协议和工具生态，让我对「浏览器 + 工具链」之间的连接有了更加直观的认识。

Chrome DevTools · MCP Server

DevTools MCP：把浏览器变成可编排的开发节点

MCP（Model Context Protocol） 是一种让工具、服务和模型之间互通的协议，而 Chrome DevTools MCP Server 则把「浏览器调试能力」提升为一个可以被调用、被编排的能力节点。 在这次 Workshop 中，我把它理解为：一扇连接 浏览器 · DevTools · AI / 工具链 的「任意门」。

我在 Workshop 中学到的关键点
DevTools MCP 是什么？
What
  • 一个把 Chrome DevTools 能力包装成 MCP Server 的实现。
  • 可以通过 MCP 调用来操控浏览器页面、元素和网络请求等。
  • 让「调试 / 自动化」和「模型 / 代理」可以说同一种语言。
能解决什么问题？
Why
  • 在 真实浏览器环境 中完成页面自动化和调试，而不是只看静态代码。
  • 把复杂重复的 DevTools 操作，交给工具 / 代理来完成。
  • 为前端 / Web 自动化场景提供更自然的「对话式」入口。
在 Workshop 里我做了什么？
Practice
  • 体验了如何通过 MCP 与 Chrome DevTools 建立连接。
  • 在浏览器页面上执行「选择元素、修改样式、截图、刷新」等操作。
  • 把这些能力与自己的 Web 开发经验结合起来，思考可以扩展的场景。
个人理解： DevTools MCP 不是一个「新的框架」，而是一种把已有浏览器调试能力 结构化、可组合、可远程调用 的方式。 当它和 GitHub Pages、CI/CD、自动化脚本甚至 AI Agent 结合时，会有很多值得挖的方向。
devtools-mcp-session.log
1 // 通过 MCP 连接 Chrome DevTools 2 invoke devtools.mcp.connect ("local-chrome") 3 await devtools.openPage ("https://odex-devfest.github.io") 4 await devtools.highlight ("#devtools-mcp") 5 await devtools.screenshot ("devtools-mcp-section.png") 6 // 通过 MCP，把页面调试过程结构化记录下来

上面是一个概念性的「伪调用」示例，用来表达这种工作流的感觉： 用一个统一的协议来调用浏览器能力。

在实际 Workshop 中，我们会基于具体的 DevTools MCP Server 实现，配置连接参数、选择页面、执行操作，并通过日志或工具界面观察结果。
