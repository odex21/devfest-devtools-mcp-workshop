Chrome DevTools · MCP Server

DevTools MCP：把浏览器变成可编排的开发节点

MCP（Model Context Protocol） 是一种让工具、服务和模型之间互通的协议，而 Chrome DevTools MCP Server 是由 Chrome 团队提供的 MCP Server，实现了对 Chrome DevTools 调试能力的封装。 简单理解：它让「浏览器调试工具」可以作为一个标准化的能力节点，被 Codex 等 MCP 客户端调用。

我在 Workshop 中学到的关键点
devtools-mcp-session.log
1 // 通过 MCP 连接 Chrome DevTools 2 invoke devtools.mcp.connect ("local-chrome") 3 await devtools.openPage ("https://odex-devfest.github.io") 4 await devtools.highlight ("#devtools-mcp") 5 await devtools.screenshot ("devtools-mcp-section.png") 6 // 通过 MCP，把页面调试过程结构化记录下来

上面是一个概念性的「伪调用」示例，用来表达这种工作流的感觉： 用一个统一的协议来调用浏览器能力。

在实际 Workshop 中，我们会基于具体的 DevTools MCP Server 实现，配置连接参数、选择页面、执行操作，并通过日志或工具界面观察结果。

下面是我在 VS Code + Codex 环境中的一段 真实配置记录，使用 TOML 把官方的 chrome-devtools-mcp Server 接入 Codex： 

# ~/.codex/config.toml 中的节选
[mcp_servers.chrome-devtools]
command = "npx"
args = ["-y", "chrome-devtools-mcp@latest"]
# 可按需加入环境变量，比如指定 Chrome 路径：
# env = { CHROME_PATH = "/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" }

配好以后，在 VS Code 里通过 Codex 面板或对话，选择 chrome-devtools 这个 MCP Server，就可以像这次 Workshop 一样： 打开本地页面、检查元素、修改样式、截全屏，甚至配合 GitHub Pages 做发布前检查。

Behind the Scenes · MCP Workflow

这一页，是怎么和 Codex 一起做出来的？

简单说：我写需求和感受，Codex GPT‑5 + Chrome DevTools MCP 在浏览器里帮我看页面、调 DevTools、改代码。

  • 启动本地 dev server，让 Codex 打开页面
  • 截图 + snapshot 看哪里「不好看」或有 bug
  • 用 DevTools MCP 在浏览器里试布局 / 样式
  • 确认 OK 后，同步改回 HTML / CSS / DaisyUI
  • 把文案抽成 Markdown，专心写 Workshop 记录
  • 收尾：About / GitHub / footer 等细节打磨

想复刻同样的流程？可以把这几个步骤当作自己的「MCP 工作流 checklist」，让 Codex 帮你搭一份属于自己的 Workshop 页面。
